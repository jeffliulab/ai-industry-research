# 产品调研模板

> 最后更新：2026-04-22

## 摘要
产品调研的粒度比公司调研更细，焦点在"**这一个产品对用户值不值、能不能赢**"。与公司调研的差别：公司调研关心"商业实体"的存续，产品调研关心"功能集合"的竞争力。

## 一、产品定位
**一句话**：`[产品名]` 是 `[谁]` 用来 `[做什么]` 的工具，它与 `[主要替代品]` 的核心差异是 `[X]`。

如果这句话写不出来，说明产品定位本身不清晰，要先停下来想清楚。

## 二、核心能力与架构
**按"用户动作"而非"功能列表"组织。**

例：不要写"支持 Markdown 编辑、支持语音输入、支持分享"，写"**写作 → 润色 → 协作 → 发布**"的完整流程。

**底层技术要点**：
- 核心模型 / 引擎
- 数据依赖
- 关键集成

**一张架构图**（可选，但强烈推荐）：使用 Mermaid 画出用户 → 界面 → 核心引擎 → 外部依赖 的关系。

## 三、版本与路线图
**时间线必含节点**：
- 首次发布
- 上一次重大版本
- 当前版本
- 已公开的下一步

**判断版本演进速度**：
- 每周更新 = 高速迭代期
- 每月更新 = 稳健期
- 每季度更新 = 成熟期或停滞期

## 四、定价与商业化
**建表**：

| 方案 | 价格 | 目标用户 | 关键限制 |
|---|---|---|---|
| Free | $0 | 个人试用 | 功能限制 |
| Pro | $X/月 | 轻度付费 | … |
| Team | $Y/seat/月 | 小团队 | … |
| Enterprise | 议价 | 大客户 | SSO / 审计 / SLA |

**关键问题**：
1. 付费墙卡在哪里？（是否足够卡而不恶心）
2. 免费版能被长期白嫖吗？
3. 企业版溢价的理由是什么？

## 五、用户反馈
**原则**：优先看**长评**、**吐槽**与**复盘**，避免"好用！"类短评。

**渠道**：
- HN / Reddit 的长评论串
- 推特 thread
- 知乎专业回答
- Substack / 个人博客
- YouTube 深度使用视频

**记录格式**：引用原文 + 来源链接 + 标明是"代表性观点"还是"少数派"。

## 六、竞品对比
**两种表法**：

**矩阵法**（能力覆盖对比）：

| 能力维度 | 本产品 | 竞品 A | 竞品 B |
|---|---|---|---|
| 核心能力 1 | ✅ | ✅ | ❌ |
| 核心能力 2 | ✅ | ❌ | ✅ |
| … | … | … | … |

**定位法**（二维坐标）：
挑两个核心维度（如"易用性 × 专业深度"），把竞品画在坐标系里。Mermaid 不擅长画这个，可以直接贴图或用文字描述象限。

## 七、我的使用笔记
**（可选但高价值。）**

真正上手用 2-4 小时，记录：
- 第一次打开最让我惊艳 / 困惑的是什么
- 我预期它能做 X，实际 Y
- 跟我 workflow 的集成难度
- 让我愿意付费 / 不愿意付费的那个瞬间

不吹不黑。没上手过就写"未使用，以下基于公开材料"。

## 八、信息源
- 官网产品页 / Changelog
- 官方文档
- 创始人或产品经理的公开演讲
- Podcast（推荐：Lenny's Newsletter、Latent Space）
- 精选用户评论

## 产品调研的常见坑
1. **把 Roadmap 当成已有功能**：明确区分"已上线"与"规划中"。
2. **用官方文档的宣传页做能力评估**：永远以**实际使用**或**用户复盘**为准。
3. **忽略定价背后的分发逻辑**：PLG / SLG / 渠道销售决定了产品形态。
4. **错把细分赢家当全局赢家**：在开发者圈好评的产品不一定在企业市场能赢。
